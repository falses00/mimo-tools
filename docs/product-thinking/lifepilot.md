# LifePilot - 深度苏格拉底式产品自问

## 1. 用户到底是谁？

LifePilot 的目标用户是那些生活节奏快、任务多样、但又不想花太多时间在任务管理工具上的人。具体来说：

- **职场人士**：每天有工作会议、项目任务、个人事务需要处理，但传统 Todo App 太繁琐
- **自由职业者**：需要同时管理多个项目、客户、个人成长，时间碎片化严重
- **学生**：课程作业、考试准备、社团活动、兼职工作交织在一起
- **创业者**：产品开发、市场推广、团队管理、个人学习都需要平衡

这些用户的共同特点是：**他们知道自己要做什么，但没有时间整理**。他们需要一个"说一句话就能记录"的工具。

## 2. 用户为什么不用普通 Todo List？

普通 Todo List 的问题在于：

1. **输入成本高**：需要打开 App → 选择分类 → 输入标题 → 设置时间 → 设置优先级 → 保存
2. **整理成本高**：完成一件事后，需要手动标记完成，然后找到下一个任务
3. **缺乏上下文**：Todo List 只是任务列表，没有"今天做了什么"的整体视角
4. **没有回顾**：大多数 Todo List 没有日报、周报功能，用户不知道自己效率如何

LifePilot 解决这些问题的方式是：**一句话输入，自动分类，自动生成日报**。

## 3. 用户为什么只想输入一句话？

因为用户在记录生活时，**思维是连续的**，不是分类的。

例如，用户想：
> "我今天写完了博客，跑了 3 公里，明天要投简历，周五前准备 AI 面试项目。"

这句话里包含了：
- 已完成（写完博客）
- 习惯（跑步 3 公里）
- 待办（投简历）
- 项目进度（准备 AI 面试项目）

如果让用户分别输入这 4 件事，他们会觉得太麻烦，最后就不记了。

**一句话输入的核心价值是：零摩擦记录**。

## 4. 系统如何判断"已完成"和"待办"？

系统使用**关键词匹配 + 上下文分析**：

**已完成关键词**：
- 完成了、写完了、做完了、跑了、学了、看完了、交了、处理了、已经、搞定

**待办关键词**：
- 要、需要、计划、准备、记得、明天、后天、周五前、下周

**判断逻辑**：
1. 如果句子包含"已完成关键词"，归类为已完成
2. 如果句子包含"待办关键词"，归类为待办
3. 如果两者都有，根据句子结构判断主语
4. 如果都不包含，归类为待办（默认）

**置信度**：
- 关键词明确匹配：0.9
- 上下文推断：0.7
- 默认归类：0.6

## 5. 系统如何处理模糊表达？

模糊表达是最大的挑战。例如：

- "今天还不错" → 无法解析，提示用户确认
- "要努力" → 归类为待办，但置信度低 (0.5)
- "跑步" → 可能是已完成，也可能是待办，需要上下文

**处理策略**：
1. **低置信度提示**：confidence < 0.6 时，在 UI 显示"需要确认"
2. **用户编辑**：用户可以修改分类、状态、优先级
3. **学习机制**：记录用户的修正，未来类似表达自动应用

## 6. 用户如何确认 AI/规则解析是否正确？

系统在解析后，会显示**结构化结果卡片**：

```
✅ 已完成
- 写完博客 [创作] [高置信度]

📋 待办事项
- 投简历 [求职] [明天] [高优先级]
- 准备 AI 面试项目 [项目] [周五前] [高优先级]

🏃 习惯记录
- 跑步 3 公里 [健康]
```

用户可以：
- 看到每项的分类、时间、优先级
- 看到置信度（高/中/低）
- 点击编辑修改任何字段
- 点击删除移除误解析项
- 点击标记完成

## 7. 误解析后如何编辑和纠正？

每个解析结果项都有**编辑按钮**：

1. **编辑标题**：点击标题直接修改
2. **改类型**：下拉选择 已完成/待办/提醒/习惯/账单
3. **改分类**：下拉选择 创作/健康/求职/项目/学习/账单/其他
4. **改优先级**：下拉选择 高/中/低
5. **改截止时间**：日期选择器
6. **删除**：确认后删除

**关键设计**：编辑是**即时生效**的，不需要额外保存步骤。

## 8. 用户如何保存历史？

保存机制有两层：

**本地保存（默认）**：
- 点击"保存记录"按钮
- 数据保存到 localStorage
- key: `mimo_lifepilot_entries`
- 刷新页面后仍存在

**API 保存（可选）**：
- 配置 PUBLIC_API_BASE_URL 后
- 数据发送到后端 API
- 保存到内存/数据库
- 支持多设备同步（未来）

## 9. 用户如何导出日报？

点击"导出日报"按钮，系统生成 Markdown 格式：

```markdown
# 今日生活记录

**日期**: 2026-04-30

## ✅ 已完成

- [x] 写完博客
- [x] 跑步 3 公里

## 📋 待办事项

- [ ] 投简历 (明天)
- [ ] 准备 AI 面试项目 (周五前)

## 🏃 习惯记录

- 跑步 3 公里

## 💡 建议

- 建议明天上午先完成投简历

---
*由 LifePilot AI 生活管家生成*
```

用户可以：
- 复制到剪贴板
- 下载为 .md 文件
- 分享到笔记软件

## 10. 用户如何知道明天该做什么？

系统在解析完成后，会自动显示**明日计划**：

```
📅 明日计划

根据您的待办事项，明天需要：
1. 投简历 [求职] [高优先级]
2. 继续准备 AI 面试项目 [项目]

💡 建议：上午先完成投简历，下午专注项目准备
```

## 11. 没有后端时是否仍可用？

**完全可用**。LifePilot 的核心功能不依赖后端：

- ✅ 自然语言解析 → 本地规则引擎
- ✅ 结果展示 → 前端渲染
- ✅ 编辑/删除/完成 → 前端状态管理
- ✅ 保存 → localStorage
- ✅ 历史 → localStorage
- ✅ 导出 → 前端生成

**后端的价值**：
- 数据同步（多设备）
- AI 增强解析（更准确）
- 持久化存储（不丢失）

## 12. 没有数据库时是否仍可用？

**完全可用**。localStorage 就是"数据库"：

- 数据保存在浏览器本地
- 刷新页面后仍存在
- 清除浏览器数据才会丢失

**数据库的价值**：
- 数据持久化（不因清除浏览器丢失）
- 多设备同步
- 复杂查询（统计、分析）

## 13. API Mode 和 Static Demo Mode 的区别是什么？

| 维度 | Static Demo Mode | API Mode |
|------|------------------|----------|
| 解析 | 本地规则引擎 | 后端 API |
| 速度 | 即时 | 网络延迟 |
| 准确性 | 基础关键词 | 可扩展 AI |
| 保存 | localStorage | API + 数据库 |
| 同步 | 无 | 多设备 |
| 离线 | 完全可用 | 需要网络 |

**UI 显示**：
- Static Demo Mode：黄色标签
- API Mode：绿色标签
- API Error Fallback：红色标签

## 14. 数据保存在哪里？

| 模式 | 保存位置 | 持久性 | 同步 |
|------|----------|--------|------|
| 本地模式 | localStorage | 浏览器内 | 无 |
| API Mode | 后端内存 | 重启丢失 | 无 |
| Supabase 模式 | 数据库 | 永久 | 多设备 |

**当前默认**：localStorage

## 15. 隐私如何保证？

1. **本地优先**：默认不上传数据
2. **用户控制**：只有点击"保存"才存储
3. **透明提示**：明确告知数据保存位置
4. **无追踪**：不收集用户行为数据
5. **开源**：代码可审查

## 16. 这个项目面试时展示什么技术点？

1. **自然语言处理**：关键词提取、上下文分析、置信度计算
2. **前端架构**：组件化、状态管理、响应式设计
3. **后端 API**：RESTful 设计、错误处理、输入校验
4. **数据持久化**：localStorage、API、数据库多层策略
5. **产品设计**：用户旅程、交互设计、中文化
6. **全栈能力**：前后端联调、部署、测试

## 19. 下一阶段如何接入真正 LLM？

**方案**：
1. 前端发送原文到 API
2. API 调用 MiMo-V2.5-Pro
3. LLM 返回结构化 JSON
4. 前端展示结果

**Fallback**：
- 无 API key → 本地规则解析
- API 超时 → 本地规则解析
- API 错误 → 本地规则解析

## 20. 下一阶段如何接入 Supabase？

**方案**：
1. 创建 Supabase 项目
2. 运行 migration
3. 配置环境变量
4. API 保存到 Supabase
5. 前端从 Supabase 读取

**Fallback**：
- 无 Supabase → localStorage
- Supabase 错误 → localStorage
- 网络问题 → localStorage

## 产品闭环图

```
用户输入一句话
    ↓
系统解析（本地规则 / API）
    ↓
显示结构化结果
    ↓
用户编辑/确认
    ↓
保存（localStorage / API）
    ↓
查看历史/时间线
    ↓
导出日报
    ↓
明日计划
```

## 用户旅程

1. **首次访问**：看到新手引导和示例
2. **输入一句话**：在文本框输入
3. **点击生成**：等待解析
4. **查看结果**：已完成、待办、习惯、账单
5. **编辑确认**：修改误解析项
6. **保存记录**：点击保存
7. **查看历史**：浏览历史记录
8. **导出日报**：生成 Markdown
9. **设置偏好**：调整工作日、提醒时间

## 输入/处理/确认/保存/复盘流程

```
输入: "我今天写完了博客，跑了3公里，明天要投简历"
    ↓
处理: 关键词匹配 → 分类推断 → 时间解析
    ↓
确认: 显示卡片 → 用户编辑 → 确认无误
    ↓
保存: localStorage / API
    ↓
复盘: 查看时间线 → 导出日报 → 明日计划
```

## 失败场景和 Fallback 策略

| 场景 | 失败原因 | Fallback |
|------|----------|----------|
| API 不可用 | 网络问题/服务器错误 | 本地规则解析 |
| 解析不准确 | 模糊表达 | 用户编辑修正 |
| 保存失败 | localStorage 满 | 提示清理空间 |
| 导出失败 | 浏览器不支持 | 复制到剪贴板 |
| 页面刷新 | 状态丢失 | localStorage 恢复 |
